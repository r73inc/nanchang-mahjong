# 2.5D DOM Gameplay Screen — Architecture Plan

**Status:** DRAFT — awaiting approval before any implementation begins.  
**Purpose:** Multi-phase blueprint for a premium Framer Motion 2.5D game table that exists alongside the existing 3D canvas. No 3D files will be touched.

---

## Background & Constraints

### What exists today

| Location                                    | Role                                                            |
| ------------------------------------------- | --------------------------------------------------------------- | ----- |
| `packages/shared/src/room.schemas.ts`       | Zod `RoomSettings` + `RoomState`; `CreateRoomInput`             |
| `packages/shared/src/game.events.ts`        | `ClientGameState`, `ClientSeatState`, socket event types        |
| `apps/api/src/rooms/dto/create-room.dto.ts` | NestJS DTO mirroring `CreateRoomInput`                          |

| `apps/web/tailwind.config.ts`               | Design tokens: `--mj-gold`, `--mj-jade-deep`, etc.              |

### What is absent today

- `viewMode` field anywhere in room/game schemas — must be added end-to-end.
- `framer-motion` package — not in `apps/web/package.json`.
- Any file under `apps/web/src/components/2d/`.

### Non-negotiable rules

1. **`apps/web/src/r3f/` is frozen.** Zero edits to any file in that directory.
2. **Shared-package edits are additive only.** New optional fields with defaults; no breaking changes to existing Zod schemas or socket types.
3. **Overlay stack is unified.** `StatusBar`, `SideRail`, `ActionToast`, `ConcedeSheet`, `ReconnectingOverlay` live above the 3D/2D fork and are rendered identically in both modes.
4. **Server remains authoritative.** No game logic in the 2D layer. All moves go through the existing `useGame` hook unchanged.
5. **One PR per phase.** Phases ship in order; each PR must pass CI before the next begins.

## Phase A — Configuration Toggle & Shared Schema

**Goal:** Propagate `viewMode: '2D' | '3D'` through the full stack so the game page can branch.  
**Touches:** `packages/shared`, `apps/api`, `apps/web/pages/room`.  
**Does NOT touch:** Any game page, game store, or 3D files.

### A1 — Shared package (`packages/shared/src/room.schemas.ts`)

Add an optional field `viewMode` to the existing `RoomSettingsSchema` Zod object with a default of `'3D'` so all existing rooms are unaffected. Because `RoomSettings` is embedded in `RoomState`, the new field propagates automatically through the `WsRoomUpdatePayload` that the frontend already receives.

Derived TypeScript type `RoomSettings` and `RoomState` will pick up `viewMode` automatically from Zod inference — no manual type edits required.

### A2 — API DTO (`apps/api/src/rooms/dto/create-room.dto.ts`)

Mirror the new field in the NestJS `CreateRoomDto` (class-validator decorator `@IsIn(['2D','3D'])`, `@IsOptional()`). The `RoomsService.create()` handler already writes `settings` verbatim into DynamoDB; no service logic changes are needed.

### A3 — Room page toggle (`apps/web/src/pages/room/room-page.tsx`)

Add a two-option toggle (radio-style button group, labelled "Table View") in the host-only settings panel that already exists in `RoomPage`. The toggle only appears before the game starts and only for the host. On change it calls the existing `updateSettings()` hook that fires `room:update-settings` — or if that socket event doesn't exist, calls `PATCH /rooms/:id/settings`. The toggle state is driven by `room.settings.viewMode`, so all four players see the current choice live via the existing `WsRoomUpdatePayload` subscription.

i18n keys needed (EN + ZH): `room.settings.viewMode`, `room.settings.viewMode.2d`, `room.settings.viewMode.3d`.

**Deliverable:** `viewMode` flows from host toggle → REST → DDB → socket broadcast → all clients. `RoomState.settings.viewMode` is now readable everywhere.

## Phase B — Framer Motion Install & Game-Page Fork

**Goal:** Install `framer-motion`; refactor `game-page.tsx` to branch between `<GameCanvas>` (3D path) and the forthcoming `<GameTable2D>` (2D path), with all overlays sitting above the fork.  
**Touches:** `apps/web/package.json`, `apps/web/src/pages/game/game-page.tsx`.  
**Does NOT touch:** Any `r3f/` file.

### B1 — Dependency install

Add `framer-motion` to `apps/web/package.json` `dependencies`. Current version constraint: `^11`. Run `pnpm install`. No other packages needed for the 2D layer.

### B2 — View-mode signal in the game page

`game-page.tsx` currently receives `gameId` from the router and loads the snapshot. The `viewMode` must be available here. Two options:

- **Option 1 (preferred):** Embed `viewMode` in the `game:snapshot` payload that the server sends on game start. This requires adding `viewMode: '2D' | '3D'` to `ClientGameState` in `packages/shared/src/game.events.ts` and to the server's `GameService.buildClientSnapshot()`. The value is copied from `room.settings.viewMode` when the game session is created.
- **Option 2 (fallback):** Store the room's `viewMode` in `sessionStorage` at the moment `WsRoomStartedPayload` arrives (before navigation), and read it back on the game page.

Option 1 is the correct long-term approach because it works across page refreshes and reconnects. Option 2 requires no shared-package change and could ship in Phase B while Option 1 catches up in a later patch.

### B3 — Fork in `game-page.tsx`

Locate the single JSX block that renders `<GameCanvas className="absolute inset-0 z-0" />` inside the `GameTable` render branch. Replace it with a conditional:

```
{viewMode === '2D' ? (
  <GameTable2D className="absolute inset-0 z-0" />
) : (
  <GameCanvas className="absolute inset-0 z-0" />
)}
```

All `z-10` and above layers (StatusBar, SeatHUD, TurnIndicator, ViewerHandHUD, SideRail, ActionToast, ConcedeSheet, ReconnectingOverlay) remain **outside and after** this conditional block — they render identically in both modes.

`<GameTable2D>` can be a stub component (renders a plain green `div`) in Phase B; subsequent phases fill it in.

### B4 — Guard the AccessibleHand

`AccessibleHand` (sr-only DOM buttons for keyboard/screen-reader users) is currently always rendered alongside the 3D canvas. Keep it as-is — it works in both modes since it's driven by the Zustand store, not by the canvas.

**Deliverable:** App runs in 3D mode by default (unchanged). Switching a room to `viewMode: '2D'` renders the green-div stub instead of the Three.js canvas. CI passes.

## Phase C — Layout Math & Felt Surface

**Goal:** Establish the pixel-space geometry for the 2D table: where each seat's discard pool, meld row, and hand live; how perspective skew is applied per-seat; and what the background looks like.  
**Touches:** New files only inside `apps/web/src/components/2d/`.  
**Does NOT touch:** `r3f/`, `table-layout.ts` (3D version — untouched).

### C1 — Pure-TS layout module

Create `apps/web/src/components/2d/layout-2d.ts` (pure functions, no React, no DOM). It exports:

- **`TABLE_ASPECT`** — locked 4:3 ratio (800×600 "design-unit" coordinate space). All positions expressed in `%` or design-units so the table scales fluidly.
- **`seatConfig(seatIdx: 0|1|2|3, viewerSeat: 0|1|2|3)`** — returns `{ role: 'bottom'|'right'|'top'|'left', transform: CSSProperties }`. The viewer's seat is always `bottom`.
- **`discardGrid(role)`** — returns `{ cols: number, rows: number, tileW: number, tileH: number, perspective: string }` for each seat position.
- **`handLayout(role)`** — returns position/size/transform for the tile row.
- **`meldLayout(role)`** — similar for open melds.

The bottom seat (viewer) gets `perspective: none` (face-on). The top seat gets a mild `perspective + rotateX(-20deg)` to suggest it faces down toward the table. Left and right seats get `rotateY(±15deg)`. This is the "2.5D" trick — subtle CSS 3D transforms applied to flat DOM zones, not to individual tiles. Individual tiles remain completely flat (no per-tile transform).

Unit tests for `seatConfig`, `discardGrid`, `handLayout` live alongside the module as `layout-2d.spec.ts`, covering all four viewer-seat rotations.

### C2 — `FeltSurface2D.tsx`

A pure CSS component that fills its container. Visual layers (bottom to top):

1. **Base** — `radial-gradient` from `var(--felt-bg)` center outward to `var(--felt-edge)`. Uses the existing `--felt-*` CSS custom properties already written by `applyTheme()` so all four felt themes work automatically.
2. **Inner vignette** — `inset` box-shadow to darken corners.
3. **Table border** — a rounded rect border using `--mj-gold` at low opacity, suggesting the table rail.
4. **Center compass rose** — a simple SVG wind-rose (East/South/West/North characters) at 30% opacity, centered. Rendered only as decorative `aria-hidden`.

No images, no external assets — pure CSS and inline SVG.

### C3 — `GameTable2D.tsx` skeleton

The root compositor. For now it renders:

```
<div className="absolute inset-0 z-0 overflow-hidden">
  <FeltSurface2D />
  {/* seat zones rendered here in later phases */}
</div>
```

Phase D–G will progressively fill in the seat zones.

**Deliverable:** Green felt table renders in 2D mode. Layout math is tested. No tiles yet.

## Phase D — Core Tile Primitive: `MahjongTile2D`

**Goal:** A single reusable tile component that renders any tile type as a styled DOM element using the existing SVG assets, with CSS effects that give it physical ceramic weight.  
**Touches:** New files inside `apps/web/src/components/2d/`.

### D1 — `MahjongTile2D.tsx` props interface

```
size: 'xs' | 'sm' | 'md' | 'lg'   — xs=opponent, sm=discard, md=meld, lg=hand
tile: TileType | 'back'             — TileType for face-up, 'back' for hidden
selected: boolean                   — viewer's selected tile state
isJing: boolean                     — jing highlight treatment
interactive: boolean                — whether pointer events fire
onSelect?: () => void
```

### D2 — Visual construction (CSS only, no SVG transforms)

Each tile is a `<button>` (interactive) or `<div>` (non-interactive) with:

- **Body:** Rounded rect (`border-radius: 4px`), `background: linear-gradient(to bottom, var(--tile-face-top), var(--tile-face-bottom))`. The `--tile-*` custom props are already written by `applyTheme()` for the three tile palettes (classic/sepia/dark).
- **Directional shadow (thickness illusion):** A thick `box-shadow` simulating a single overhead light source. The shadow X/Y offsets are **not hardcoded** — they are computed per-tile based on the seat `role` prop so the implied light source is physically uniform across the whole board. The target screen-space shadow vector is `(+2px, +6px)` (light from top-left, shadow down-right). Because tile containers are CSS-rotated per seat, the tile-local offsets must be inverse-rotated to compensate:
  - `bottom` (0° rotation): tile-local `(+2px, +6px)` → appears as `(+2, +6)` in screen space ✓
  - `right` (container −90°): tile-local `(−6px, +2px)` → rotated −90° appears as `(+2, +6)` ✓
  - `top` (container 180°): tile-local `(−2px, −6px)` → rotated 180° appears as `(+2, +6)` ✓
  - `left` (container +90°): tile-local `(+6px, −2px)` → rotated +90° appears as `(+2, +6)` ✓
    Shadow thickness (the `0` spread parameter) scales with tile size. This single directional shadow is the primary "2.5D ceramic weight" effect.
- **Face stamp:** An `<img>` tag pointing to `tileTexturePath(tile)` (the same helper already used in `ViewerHandHUD`), with `width/height 80%` centered via absolute positioning.
- **Selected state:** `box-shadow` expands upward (`translateY(-6px)`) + gold ring border. Animated with Framer Motion `animate` prop.
- **Jing state:** A gold emissive ring (`box-shadow: 0 0 12px var(--mj-gold)`) + the `节` label below the tile, matching the 3D treatment visually.
- **Back face:** Renders `Back.svg` (existing asset); body uses `--tile-back-bg` custom prop.

### D3 — Sizing table (design-unit px at 800w reference)

| Size | Width | Height | Bottom-shadow |
| ---- | ----- | ------ | ------------- |
| `xs` | 28px  | 38px   | 4px           |
| `sm` | 36px  | 48px   | 5px           |
| `md` | 44px  | 60px   | 6px           |
| `lg` | 56px  | 76px   | 8px           |

All sizes scale via CSS `font-size`-based `em` units or CSS container queries so the table is fully fluid.

### D4 — Accessibility

`aria-label` on each tile: same pattern as the existing DOM `MahjongTile` component (`"3 of Characters"`, `"Red Dragon"`, etc.). Interactive tiles get `role="button"`. Non-interactive tiles get `role="img"`. Back-face tiles get `aria-label="Hidden tile"`.

### D5 — Motion

`MahjongTile2D` wraps its outermost element in Framer Motion's `motion.div`. Specific animation variants (draw, discard, claim) are defined here as named `variants` and triggered by parent orchestrators in Phase G. The `reduced-motion` media query disables all variants — a single CSS rule in `index.css` sets `--fm-duration: 0` which Framer Motion respects via its `MotionConfig` provider.

**Deliverable:** `MahjongTile2D` is a standalone, fully styled tile in all four sizes. Storybook-style dev test: temporarily drop `<MahjongTile2D tile="Man1" size="lg" />` into a route to visually verify before moving on.

## Phase E — Player Hand: Drag-to-Sort with Framer Motion

**Goal:** The viewer's hand at the bottom of the screen — oversized, fully draggable for manual sorting, tap-to-select-then-discard.  
**Touches:** `apps/web/src/components/2d/PlayerHand2D.tsx`.

### E1 — Architecture decision: local order state

The server sends `hand: TileType[]` in a fixed server-side order. The 2D hand layer maintains its own `localOrder: TileType[]` in component state, initialized from the snapshot hand and updated on drag completion. When the server sends a new snapshot (draw, claim), the new tile is appended to the end of `localOrder` and existing tiles stay in their user-sorted positions via a stable identity merge. On discard, the tile is removed from `localOrder`. This is purely cosmetic — the Zustand store and socket events remain unaffected.

### E2 — `PlayerHand2D.tsx` structure

Uses `Reorder.Group` (horizontal axis) from Framer Motion wrapping `Reorder.Item` for each tile. Spring physics config: `stiffness: 400, damping: 30` — snappy but not instant, giving a "card sliding on felt" feel. While dragging, the dragged tile lifts by `translateY(-12px)` and the gap opens with a spring-animated placeholder.

The entire hand is positioned at the bottom of `GameTable2D`, intentionally oversized (`lg` size tiles) with a shallow `perspective` applied to the container so the row reads as "close to camera." No individual tile perspective — the row-level container carries the effect.

### E3 — Tap-to-select → discard flow

1. Tap a tile → `selectTile(idx)` in Zustand store (existing action). Selected tile animates up by 8px.
2. Tap selected tile again → `onDiscard(tile)` callback (wired to existing `useGame` hook's `discard()` function).
3. Alternatively, after selecting, tapping elsewhere deselects.
4. During the claim window (opponent's discard available), the hand greys out and drag is disabled; tap fires a claim selection if applicable.

### E4 — Pending discard visual

When `snapshot.pendingDiscard` is set (the viewer drew and must discard), a pulsing gold ring appears around the most recently drawn tile (last in `localOrder`) and a tooltip "Tap to discard" shows beneath the hand.

### E5 — Tests

- `localOrder` merge logic: unit test for new-tile append and stable-sort across draws.
- Drag reorder: Framer Motion's test utilities with `@testing-library/user-event`.
- Discard flow: spy on `useGame.discard()` when tile is double-tapped.

**Deliverable:** Viewer's hand fully interactive. Manual reorder does not affect game state. Discard works end-to-end in 2D mode.

## Phase F — Opponent Hands, Discard Pools, Open Melds

**Goal:** All four seat zones rendered correctly: face-down opponent hands, discard history grids, open meld rows.  
**Touches:** `apps/web/src/components/2d/` — four new components.

### F1 — `OpponentHand2D.tsx`

Renders `handCount` face-down tiles for one opponent seat. Positioned per `handLayout(role)` from `layout-2d.ts`. For `top` and `left/right` seats, the tile row is rotated via CSS `transform: rotate(Xdeg)` on the container to face "inward" toward the table center — same rotation angles as the perspective hints from Phase C. Individual tiles use `size="xs"` with `tile="back"`. AFK state shows a grey overlay. `afk` status drawn from `ClientSeatState.afk`.

### F2 — `DiscardPool2D.tsx`

Renders `discards: TileType[]` for one seat as a grid. Grid spec from `discardGrid(role)` in `layout-2d.ts`. Layout rules matching the visual convention in reference image 18564.jpg:

- Tiles flow in reading order (left-to-right, top-to-bottom) for the bottom seat.
- For left/right seats the grid is rotated 90° on the container; tiles read top-to-bottom.
- For the top seat the grid is rotated 180°.
- Last discard (index `discards.length - 1`) receives a "last discard" pulse if `snapshot.discardedBySeat === thisSeatIdx` and the claim window is open — same gold pulse animation used in the existing DOM overlay.
- Tile size: `sm`. No drag; click does nothing (read-only history).

### F3 — `OpenMelds2D.tsx`

Renders `openMelds: Meld[]` for one seat. Each meld (pung/kong/chow) is a tightly spaced horizontal group of three or four tiles. Melds stack left-to-right for bottom seat; rotate with container for other seats. Tile size: `md`. Kong bonus tile sits offset above the group at `translateY(-6px)`. No interaction.

### F4 — `SeatLabel2D.tsx`

A nameplate chip showing player handle, wind character, score, AFK badge, and connectivity dot. Mirrors `SeatHUD`'s Nameplate component aesthetically but is positioned by `GameTable2D` at the corner of each seat zone rather than at CSS compass corners. This means the label sits adjacent to that seat's tile hand rather than at the screen edge — closer to the reference images. Uses existing design tokens; no new CSS variables.

### F5 — `GameTable2D.tsx` with all zones

The compositor lays out all components using CSS Grid (a 3×3 grid of areas: top-left/top/top-right, left/center/right, bottom-left/bottom/bottom-right). The center cell holds `FeltSurface2D`. Seat zones (`OpponentHand2D`, `DiscardPool2D`, `OpenMelds2D`, `SeatLabel2D`) are positioned in their respective CSS grid areas using the `role` derived from `viewerSeat` + `seatIdx`.

The `snapshot` is read directly from `useGameStore` (same pattern as `GameCanvas.tsx` uses `subscribeWithSelector`). No prop-drilling; each child reads only its seat's `ClientSeatState`.

**Deliverable:** Full static board renders in 2D mode. Opponents, discards, and melds visible. No animations yet; no interaction beyond Phase E hand.

## Phase G — Framer Motion Animations & Event Reactions

**Goal:** Bring the board to life with draw animations, discard flights, and claim flashes — all coordinated by Framer Motion's `AnimatePresence` and layout animations.  
**Touches:** Only files under `apps/web/src/components/2d/`; one read of game event socket in `game-page.tsx` to trigger 2D-only animation callbacks.

### G1 — Animation vocabulary

All animations respect `prefers-reduced-motion` via a single `MotionConfig` wrapper at the `GameTable2D` root: `{ reducedMotion: 'user' }`. When reduced-motion is active, all variants collapse to instant opacity cuts.

| Event                  | Visual                                                                                                     |
| ---------------------- | ---------------------------------------------------------------------------------------------------------- |
| Draw                   | New tile in player hand fades in from `opacity:0, scale:0.8` over 200ms                                    |
| Discard (self)         | Tile shrinks and flies from hand row to discard pool via Framer Motion `layoutId` shared element animation |
| Discard (opponent)     | A brief pulse ring appears on the last tile in opponent's discard pool                                     |
| Pung/Chow/Kong claimed | Three tiles in opponent's hand collapse together and fly to their new open meld zone via `layoutId`        |
| Claim window open      | `DiscardPool2D`'s last tile glows gold (reuses existing `last-discard-pulse` keyframe)                     |
| Win                    | All tiles in viewer's hand animate to a fan-spread before the `GameEndScreen` takes over                   |

### G2 — `layoutId` strategy

Framer Motion's `layoutId` allows a tile element to animate between two positions in the DOM tree. Because `TileType` values are bare strings with no stable identity, using raw indices would cause Framer Motion to misidentify tiles during a drag-while-discarding scenario. Instead:

**`localOrder` stores wrapped objects, not bare strings.** When initialising (or updating on draw), each tile entry is `{ id: nanoid(), tile: TileType }`. The `id` is an ephemeral unique identifier that lives only in component state — it never reaches the server. When a new tile is drawn from the server snapshot, it is appended as `{ id: nanoid(), tile: newTile }`. Existing entries keep their ids. On discard, the matching entry is removed by `id`.

The `layoutId` for each tile in `PlayerHand2D` is `hand-${entry.id}`. The `layoutId` for the corresponding entry in `DiscardPool2D` is also derived from this same `id`, threaded via a ref passed from the hand to the pool. When a discard fires, Framer Motion sees a `layoutId` disappear from the hand and reappear in the pool and smoothly animates the transition regardless of how many tiles of the same `TileType` exist in the hand.

### G3 — `AnimatePresence` placements

- `PlayerHand2D` — wrap the `Reorder.Group` children in `AnimatePresence mode="popLayout"` so drawn tiles animate in and discarded tiles animate out without layout jumps.
- `DiscardPool2D` — wrap each tile row in `AnimatePresence` so new discards fly in.
- `OpenMelds2D` — wrap each meld group in `AnimatePresence` so newly formed melds animate in as a group.

### G4 — Minimal game-page wiring

`GameTable2D` subscribes to `snapshot` from Zustand. When `snapshot.phase` transitions from `'awaiting_claims'` to `'playing'` after a claim, the component derives which meld was just formed and triggers the corresponding animation variant. No new socket subscriptions — all derived from snapshot diffs, consistent with the existing approach in `GameCanvas`.

**Deliverable:** Board feels alive. Draw, discard, and claim actions have motion feedback. All motion is disabled cleanly under `prefers-reduced-motion`.

## Phase H — Integration Polish, A11y & Tests

**Goal:** Full integration pass, accessibility audit, and test coverage to ship-ready standard.  
**Touches:** `apps/web/src/components/2d/`, `apps/web/src/pages/game/game-page.tsx`, `apps/web/src/index.css`.

### H1 — `prefers-reduced-motion` global CSS

Add to `apps/web/src/index.css`:

@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}

This is Phase 12B's A11y deliverable and belongs here so both 3D and 2D modes respect it.
