# 2.5D Game Table — Bug Log

Active bug log for the 2.5D game table.

---

## Open Bugs

### BUG-2D-01 — Claimed discard tile remains visible in the discard pool

**Status:** FIXED — PR #50
**Root cause:** `pung()`, `chow()`, and `kongFromDiscard()` in `packages/engine/src/engine.ts`
only patched the _claiming_ seat (adding the open meld) but never removed the claimed tile
from the _discarder's_ `discards` array. Because `toClientSnapshot` reads `seat.discards`
verbatim, the ghost tile appeared in every client snapshot until the hand ended.
**Fix:** Added `patchTwoSeats()` private helper to `GameEngine`. All three claim methods
now simultaneously patch the claiming seat (meld) and the discarder (slice last tile from
`discards`).
**Tests added:** 3 new engine regression tests in `engine.test.ts`:

- `pung removes the claimed tile from the discarder discards array`
- `chow removes the claimed tile from the discarder discards array`
- `kongFromDiscard removes the claimed tile from the discarder discards array`
  **Files changed:** `packages/engine/src/engine.ts`, `packages/engine/src/__tests__/engine.test.ts`

### ~~BUG-2D-02~~ — Layout scaling is viewport-unaware; tiles overflow on smaller screens and after melds

**Status:** FIXED — PR #51
**Root causes:**

1. `TILE_DIMS` pixel values in `MahjongTile2D` were hardcoded — tiles never shrank regardless of viewport.
2. CSS Grid used fixed `22% / 56% / 22%` rows. Viewer zone overflowed when melds were added.
3. Left/right opponent tiles rotate 90° — their visual width = center row HEIGHT. Fixed-pixel tiles overflowed when center row was shorter than reference.
4. Viewer zone was constrained to 56% center column, inadequate on narrow/mobile viewports.

**Fix (PR #51):**

- `Table2DContext.tsx` (new) — `computeTileScale(w,h)` derives scale in [0.25, 1.0] from two constraints (opponent visual width = 50% tableH; viewer width = full tableW). Context broadcasts to all descendant `MahjongTile2D` instances.
- `GameTable2D.tsx` — `ResizeObserver` tracks actual pixel dimensions. Grid rows → `minmax(0,1fr) / minmax(0,2fr) / auto` (viewer zone auto-sizes). Bottom area spans all 3 columns (`"bottom bottom bottom"`) → viewer hand gets full table width.
- `MahjongTile2D.tsx` — reads `tileScale` from context; scales all pixel dims (w, h, border-radius, shadows, font-size, lift-Y).
- `test-setup.ts` — `ResizeObserver` stub added (800×600 → tileScale=1.0).
- 6 new `computeTileScale` unit tests in `GameTable2D.test.tsx`.

**Files changed:** `Table2DContext.tsx` (new), `GameTable2D.tsx`, `MahjongTile2D.tsx`, `test-setup.ts`, `GameTable2D.test.tsx`

### ~~BUG-2D-03~~ — Discarded tiles appear in each seat's zone rather than the centre of the table

**Status:** FIXED — PR #52
**Symptom:** Each player's discarded tiles were rendered inside that player's own seat
zone (the 22% or 56% grid area), not in the shared centre area of the table. This made
the table feel like four isolated panels rather than a communal playing surface.
**Root cause:** The four `DiscardPool2D` instances were children of each seat zone
container. Seat zones are CSS-rotated (`rotateZ(±90°/180°)`) and occupy the side/top
strips of the grid, so discards appeared in the seat strips rather than the felt centre.
**Fix (PR #52):**

- `CenterDiscards2D.tsx` (new) — renders all four `DiscardPool2D` instances with
  `gridArea: 'center'` and `position: relative`. Each pool is absolutely positioned
  near its seat's edge (top → top edge centred; right → right edge centred; etc.).
  All tiles receive `tileRole="bottom"` so the box-shadow direction is correct for the
  un-rotated context (no inverse-rotation compensation needed).
- `layout-2d.ts` — `discardGrid()` now returns role-specific column counts:
  `top`/`bottom` → 6 cols (wide horizontal strip); `left`/`right` → 3 cols (narrow
  vertical strip, since the pool is no longer inside a rotated container).
- `DiscardPool2D.tsx` — added optional `tileRole?: SeatRole` prop (defaults to `role`)
  so the layout role (grid columns) and shadow role (tile rendering) can differ.
- `GameTable2D.tsx` — removed `DiscardPool2D` from both viewer zone and opponent
  zones; added `<CenterDiscards2D viewerSeat={viewerSeat} />` after the felt layer.
- `layout-2d.spec.ts` — updated the "all roles share same spec" assertion to two new
  tests reflecting the split col counts.

**Files changed:** `CenterDiscards2D.tsx` (new), `GameTable2D.tsx`, `DiscardPool2D.tsx`,
`layout-2d.ts`, `layout-2d.spec.ts`

### ~~BUG-2D-04~~ — Opponent hidden hands appear at the outer screen edge; open melds disconnected from hand

**Status:** FIXED — PR #53
**Symptom:** Opponent face-down hand tiles appear at the extreme outer edge of their seat
zone (top opponent's tiles at the very top of the screen; left/right opponents' tiles at
the left/right screen edges). Open melds appear near the inner (centre-facing) edge of the
same zone, leaving a large gap between the hand and the melds. At game start (no melds
yet) the hand may be pushed completely off-screen on mobile.
**Root cause — CSS rotation inverts flex child visual order:**
Each opponent seat zone container has a `rotateZ` transform applied
(`rotateZ(180deg)` for top, `rotateZ(±90deg)` for left/right). The flex layout is computed
in CSS space _before_ rotation is applied, but visually the container is flipped. With
`justifyContent: 'space-between'` and `flexDirection: 'column'`:

| Seat         | CSS first child (SeatLabel)         | CSS last child (OpponentHand / OpenMelds) |
| ------------ | ----------------------------------- | ----------------------------------------- |
| top (180°)   | visually → inner edge (near centre) | visually → outer edge (top of screen)     |
| right (−90°) | visually → inner edge               | visually → outer edge (right of screen)   |
| left (+90°)  | visually → inner edge               | visually → outer edge (left of screen)    |

So the hand (last child) ends up at the OUTER edge and the label ends up at the INNER
edge — the opposite of the intended layout. When melds are added as a third child, the
hand moves to the visual middle while melds take the outer edge, but the hand and melds
are still separated by most of the zone height.

**Fix (PR #53):**
Two changes in `GameTable2D.tsx`:

1. **DOM child order reversed to compensate for rotation.** For all three rotations the
   DOM-first child ends up at the visual inner edge. `SeatLabel2D` is now the _last_ DOM
   child so rotation puts it at the outer (perimeter) edge.
2. **Hand and melds grouped in a sub-container.** `OpenMelds2D` + `OpponentHand2D` are
   wrapped in a nested flex div so they stay adjacent regardless of whether melds exist.
   `OpenMelds2D` is DOM-first within the sub-group so it sits closest to the compass rose
   (matching real-table layout where open melds face the centre). When no melds exist
   `OpenMelds2D` returns null and the hand alone anchors to the inner edge — never
   pushed off-screen.

**Files changed:** `apps/web/src/components/2d/GameTable2D.tsx`

### ~~BUG-2D-05~~ — Discards should be a single combined centre pile, not four per-player pools

**Status:** FIXED — PR #53
**Symptom:** After BUG-2D-03's fix, all discards appear inside the `center` CSS grid cell
but as four _separate_ per-player pools anchored near each seat's edge (top pool near top
edge, right pool near right edge, etc.). The user wants all discards merged into one
combined pile at the true centre of the felt — matching how real Mahjong tables look from
above, where all tiles flow into a single shared pool.
**Root cause:** `CenterDiscards2D` renders four independent `DiscardPool2D` instances
positioned near their seat's edge. There is no mechanism to merge tiles from all four
seats into one grid.
**Fix (PR #53):**

- `CombinedDiscardPool2D.tsx` (new) — merges all four seats' discards using option (b)
  round-robin interleave (`seat0[0], seat1[0], seat2[0], seat3[0], seat0[1], …`).
  Rendered as a single 8-column grid absolutely centred in the centre cell
  (`top:50%; left:50%; transform:translate(-50%,-50%)`).
  Preserves the gold pulse on `snapshot.discardedBySeat`'s last tile and the
  `DiscardContext` `layoutId` on the viewer's most recent discard for the flight animation.
- `CenterDiscards2D.tsx` — simplified to a thin `position:relative` wrapper that renders
  `CombinedDiscardPool2D`; no longer needs `viewerSeat` prop.
- `GameTable2D.tsx` — `<CenterDiscards2D>` call loses the `viewerSeat` prop.
- `GameTable2D.test.tsx` — per-seat `discard-pool-N` testId assertions replaced with
  3 new tests using `combined-discard-pool`.

**Files changed:** `CombinedDiscardPool2D.tsx` (new), `CenterDiscards2D.tsx`,
`GameTable2D.tsx`, `GameTable2D.test.tsx`

### BUG-2D-06 — No back-button intercept; no in-progress rejoin path

**Status:** FIXED — PR #55
**Root cause:**

1. `game-page.tsx` had no popstate / React Router `useBlocker` guard — browser-back fired instantly without confirmation.
2. No rejoin UI existed; a full page navigation destroyed the React tree with no re-entry point.
   **Fix:** Added `useBlocker` in `GamePage` (fires while `phase` is `playing` or `awaiting_claims`); `LeaveGameSheet` (z-60 fixed overlay) with "Stay in Game" / "Leave" buttons; `localStorage` key `mj:active-game` persists the active game id; `LobbyPage` reads the key on mount and shows a "Game in Progress" rejoin card with "Rejoin Game" and dismiss buttons.
   **Files changed:** `game-page.tsx`, `lobby-page.tsx`, `en.json`, `zh.json`, `game-page.test.tsx`, `lobby-page.test.tsx`

## Key Learnings

- **Engine claim methods must patch two seats.** When a claim is resolved the claimed tile
  leaves the discarder's pile and enters the claimer's meld — the engine must update both
  seats atomically in one `withState` call using a two-seat patch. Patching only the
  claimer leaves a ghost entry in the discarder's `discards` array that propagates through
  every subsequent `toClientSnapshot` broadcast.

- **CSS rotateZ reverses DOM child visual order in a flex container.** For all three
  rotations used (180°, +90°, −90°) the DOM-FIRST flex child ends up at the visual
  INNER edge (facing the table centre) and the DOM-LAST child ends up at the visual
  OUTER edge (screen perimeter). This is the opposite of the intuitive reading order.
  When laying out a rotated opponent zone always put centre-facing content (hand, melds)
  first in the DOM and edge labels last.

- **CSS-rotated seat zone containers break the spatial meaning of "inside".** Rendering
  discard pools as children of a `rotateZ(±90°/180°)` seat zone makes them appear in the
  seat strip, not the felt centre. The fix is to lift the pools out of the rotated
  containers into a dedicated un-rotated overlay in the `center` grid area and position
  them absolutely per role. As a side effect, tile shadow offsets must be recalculated for
  the un-rotated context (use `role="bottom"` for all centre-area tiles rather than each
  seat's inverse-rotated offset). Similarly, column counts change: left/right pools that
  were 6-wide-and-visually-rotated become 3-wide-and-upright.
