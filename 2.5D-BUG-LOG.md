## Open Bugs

### BUG-2D-01 — Claimed discard tile remains visible in the discard pool

**Status:** FIXED — PR #50
**Root cause:** `pung()`, `chow()`, and `kongFromDiscard()` in `packages/engine/src/engine.ts`
only patched the _claiming_ seat (adding the open meld) but never removed the claimed tile
from the _discarder's_ `discards` array. Because `toClientSnapshot` reads `seat.discards`
verbatim, the ghost tile appeared in every client snapshot until the hand ended.
**Fix:** Added `patchTwoSeats()` private helper to `GameEngine`. All three claim methods
now simultaneously patch the claiming seat (meld) and the discarder (slice last tile from
`discards`).
**Tests added:** 3 new engine regression tests in `engine.test.ts`:

- `pung removes the claimed tile from the discarder discards array`
- `chow removes the claimed tile from the discarder discards array`
- `kongFromDiscard removes the claimed tile from the discarder discards array`
  **Files changed:** `packages/engine/src/engine.ts`, `packages/engine/src/__tests__/engine.test.ts`

### ~~BUG-2D-02~~ — Layout scaling is viewport-unaware; tiles overflow on smaller screens and after melds

**Status:** FIXED — PR #51
**Root causes:**

1. `TILE_DIMS` pixel values in `MahjongTile2D` were hardcoded — tiles never shrank regardless of viewport.
2. CSS Grid used fixed `22% / 56% / 22%` rows. Viewer zone overflowed when melds were added.
3. Left/right opponent tiles rotate 90° — their visual width = center row HEIGHT. Fixed-pixel tiles overflowed when center row was shorter than reference.
4. Viewer zone was constrained to 56% center column, inadequate on narrow/mobile viewports.

**Fix (PR #51):**

- `Table2DContext.tsx` (new) — `computeTileScale(w,h)` derives scale in [0.25, 1.0] from two constraints (opponent visual width = 50% tableH; viewer width = full tableW). Context broadcasts to all descendant `MahjongTile2D` instances.
- `GameTable2D.tsx` — `ResizeObserver` tracks actual pixel dimensions. Grid rows → `minmax(0,1fr) / minmax(0,2fr) / auto` (viewer zone auto-sizes). Bottom area spans all 3 columns (`"bottom bottom bottom"`) → viewer hand gets full table width.
- `MahjongTile2D.tsx` — reads `tileScale` from context; scales all pixel dims (w, h, border-radius, shadows, font-size, lift-Y).
- `test-setup.ts` — `ResizeObserver` stub added (800×600 → tileScale=1.0).
- 6 new `computeTileScale` unit tests in `GameTable2D.test.tsx`.

**Files changed:** `Table2DContext.tsx` (new), `GameTable2D.tsx`, `MahjongTile2D.tsx`, `test-setup.ts`, `GameTable2D.test.tsx`

### ~~BUG-2D-03~~ — Discarded tiles appear in each seat's zone rather than the centre of the table

**Status:** FIXED — PR #52
**Symptom:** Each player's discarded tiles were rendered inside that player's own seat
zone (the 22% or 56% grid area), not in the shared centre area of the table. This made
the table feel like four isolated panels rather than a communal playing surface.
**Root cause:** The four `DiscardPool2D` instances were children of each seat zone
container. Seat zones are CSS-rotated (`rotateZ(±90°/180°)`) and occupy the side/top
strips of the grid, so discards appeared in the seat strips rather than the felt centre.
**Fix (PR #52):**

- `CenterDiscards2D.tsx` (new) — renders all four `DiscardPool2D` instances with
  `gridArea: 'center'` and `position: relative`. Each pool is absolutely positioned
  near its seat's edge (top → top edge centred; right → right edge centred; etc.).
  All tiles receive `tileRole="bottom"` so the box-shadow direction is correct for the
  un-rotated context (no inverse-rotation compensation needed).
- `layout-2d.ts` — `discardGrid()` now returns role-specific column counts:
  `top`/`bottom` → 6 cols (wide horizontal strip); `left`/`right` → 3 cols (narrow
  vertical strip, since the pool is no longer inside a rotated container).
- `DiscardPool2D.tsx` — added optional `tileRole?: SeatRole` prop (defaults to `role`)
  so the layout role (grid columns) and shadow role (tile rendering) can differ.
- `GameTable2D.tsx` — removed `DiscardPool2D` from both viewer zone and opponent
  zones; added `<CenterDiscards2D viewerSeat={viewerSeat} />` after the felt layer.
- `layout-2d.spec.ts` — updated the "all roles share same spec" assertion to two new
  tests reflecting the split col counts.

**Files changed:** `CenterDiscards2D.tsx` (new), `GameTable2D.tsx`, `DiscardPool2D.tsx`,
`layout-2d.ts`, `layout-2d.spec.ts`

### ~~BUG-2D-04~~ — Opponent hidden hands appear at the outer screen edge; open melds disconnected from hand

**Status:** FIXED — PR #53
**Symptom:** Opponent face-down hand tiles appear at the extreme outer edge of their seat
zone (top opponent's tiles at the very top of the screen; left/right opponents' tiles at
the left/right screen edges). Open melds appear near the inner (centre-facing) edge of the
same zone, leaving a large gap between the hand and the melds. At game start (no melds
yet) the hand may be pushed completely off-screen on mobile.
**Root cause — CSS rotation inverts flex child visual order:**
Each opponent seat zone container has a `rotateZ` transform applied
(`rotateZ(180deg)` for top, `rotateZ(±90deg)` for left/right). The flex layout is computed
in CSS space _before_ rotation is applied, but visually the container is flipped. With
`justifyContent: 'space-between'` and `flexDirection: 'column'`:

| Seat         | CSS first child (SeatLabel)         | CSS last child (OpponentHand / OpenMelds) |
| ------------ | ----------------------------------- | ----------------------------------------- |
| top (180°)   | visually → inner edge (near centre) | visually → outer edge (top of screen)     |
| right (−90°) | visually → inner edge               | visually → outer edge (right of screen)   |
| left (+90°)  | visually → inner edge               | visually → outer edge (left of screen)    |

So the hand (last child) ends up at the OUTER edge and the label ends up at the INNER
edge — the opposite of the intended layout. When melds are added as a third child, the
hand moves to the visual middle while melds take the outer edge, but the hand and melds
are still separated by most of the zone height.

**Fix (PR #53):**
Two changes in `GameTable2D.tsx`:

1. **DOM child order reversed to compensate for rotation.** For all three rotations the
   DOM-first child ends up at the visual inner edge. `SeatLabel2D` is now the _last_ DOM
   child so rotation puts it at the outer (perimeter) edge.
2. **Hand and melds grouped in a sub-container.** `OpenMelds2D` + `OpponentHand2D` are
   wrapped in a nested flex div so they stay adjacent regardless of whether melds exist.
   `OpenMelds2D` is DOM-first within the sub-group so it sits closest to the compass rose
   (matching real-table layout where open melds face the centre). When no melds exist
   `OpenMelds2D` returns null and the hand alone anchors to the inner edge — never
   pushed off-screen.

**Files changed:** `apps/web/src/components/2d/GameTable2D.tsx`

### ~~BUG-2D-05~~ — Discards should be a single combined centre pile, not four per-player pools

**Status:** FIXED — PR #53
**Symptom:** After BUG-2D-03's fix, all discards appear inside the `center` CSS grid cell
but as four _separate_ per-player pools anchored near each seat's edge (top pool near top
edge, right pool near right edge, etc.). The user wants all discards merged into one
combined pile at the true centre of the felt — matching how real Mahjong tables look from
above, where all tiles flow into a single shared pool.
**Root cause:** `CenterDiscards2D` renders four independent `DiscardPool2D` instances
positioned near their seat's edge. There is no mechanism to merge tiles from all four
seats into one grid.
**Fix (PR #53):**

- `CombinedDiscardPool2D.tsx` (new) — merges all four seats' discards using option (b)
  round-robin interleave (`seat0[0], seat1[0], seat2[0], seat3[0], seat0[1], …`).
  Rendered as a single 8-column grid absolutely centred in the centre cell
  (`top:50%; left:50%; transform:translate(-50%,-50%)`).
  Preserves the gold pulse on `snapshot.discardedBySeat`'s last tile and the
  `DiscardContext` `layoutId` on the viewer's most recent discard for the flight animation.
- `CenterDiscards2D.tsx` — simplified to a thin `position:relative` wrapper that renders
  `CombinedDiscardPool2D`; no longer needs `viewerSeat` prop.
- `GameTable2D.tsx` — `<CenterDiscards2D>` call loses the `viewerSeat` prop.
- `GameTable2D.test.tsx` — per-seat `discard-pool-N` testId assertions replaced with
  3 new tests using `combined-discard-pool`.

**Files changed:** `CombinedDiscardPool2D.tsx` (new), `CenterDiscards2D.tsx`,
`GameTable2D.tsx`, `GameTable2D.test.tsx`

### BUG-2D-06 — No back-button intercept; no in-progress rejoin path

**Status:** OPEN
**Symptom:** On mobile (and desktop) pressing the browser/OS back button during an
active game immediately navigates to the lobby. There is no confirmation dialog.
A player who fat-fingers back, or whose browser navigates away on a phone call /
app switch, has no way to return to their in-progress game — they land in the lobby
and see only "Create room" and "Join room" with no "Rejoin" affordance.
**Root cause:**

1. `game-page.tsx` has no `beforeunload` / `popstate` / React Router `useBlocker`
   guard, so browser-back fires instantly without confirmation.
2. No "rejoin in-progress game" UI exists anywhere. The socket `useGame` hook
   reconnects transparently if the _socket_ drops, but a full page navigation
   destroys the React tree — there is no entry point back.
   **Proposed fix:**

- Add a React Router `useBlocker` in `GamePage` that triggers a confirmation sheet
  ("Leave game? You can rejoin from the lobby.") when `snapshot.phase` is `playing`
  or `awaiting_claims`. On mobile the OS back gesture fires the same popstate event
  that `useBlocker` intercepts.
- Add a "Rejoin" card in `LobbyPage` / `HomePage` that appears when the API reports
  an active game session for the current user. `GET /rooms/active` (or a small
  extension to an existing endpoint) can return the active `gameId` if one exists.
  **Files affected:** `game-page.tsx`, `lobby-page.tsx` (or `home-page.tsx`),
  possibly a new API endpoint `GET /rooms/active`.

### BUG-2D-07 — Mobile: opponent hands invisible or rendered as tiny off-screen columns

**Status:** OPEN
**Screenshot evidence (portrait, 375×812 px device):**

- **Top opponent:** Zero tiles visible — only a nameplate score badge at the very top.
- **Left opponent:** Zero tiles visible — only a "● 0" score badge at the left edge.
- **Right opponent:** 13 tiny red squares (~8×10 px each) appear as a vertical column
  jammed against the right edge of the screen, partially obscured by the history
  toggle tab (≡ button).
- In **landscape** orientation all three opponent tile columns become partially visible
  as small red strips along each edge, but still only ~10–15 px tall per tile.
  **Root cause (confirmed by screenshots):**
  The current layout assigns `22%` CSS Grid columns to the left and right seats. On a
  375 px portrait phone that is only ~82 px. Left/right seat zones are `rotateZ(±90°)`,
  so their visual _width_ = the _height_ of the `center` grid row. The center row uses
  `minmax(0, 2fr)` and on portrait mobile that row height is very small (squeezed
  between the ~70 px status bar and the `auto` bottom zone that the viewer's large
  hand occupies). The rotated tile row therefore renders into a near-zero-height area
  and overflows upward/downward out of view. `overflow: hidden` is NOT set on the seat
  zones (correct, to allow selected tile lift), but the containing grid row is so
  small the tiles are simply outside the visual port. `computeTileScale` starts from a
  default `800×600` stub and would only help if the tiles fit inside the measured
  container — which collapses to ~0 px before the `ResizeObserver` fires.
  The top opponent zone occupies the `top` grid area (`.  top  .` template row) but on
  portrait mobile that entire row collapses because the left/right columns are 22% of
  a narrow screen, leaving virtually no height budget for the top strip.
  **Root cause summary:** Portrait mobile has too little center-row height for the
  rotated 90° tile rows to render visibly. The `22/56/22 %` column split is a
  desktop-first assumption that breaks completely on narrow portrait screens.
  **Proposed fix:** Full mobile layout redesign for `GameTable2D`. On viewports narrower
  than 600 px (portrait phones), switch to a **compact 3-row stacked layout** instead
  of the 3-column CSS Grid:
- **Row 1 (top, ~60 px):** Top opponent — horizontal strip of face-down tile count
  badge (e.g. `🀫 ×13`) + nameplate. No individual tiles.
- **Row 2 (center, flex-grow):** Left strip | felt+discard pool | right strip,
  where left/right strips are narrow (40–50 px) vertical count badges, not rotated
  tile rows.
- **Row 3 (bottom, auto):** Viewer's open melds + `PlayerHand2D` (unchanged).
  Individual face-down tiles for opponents are decorative on mobile — a tile-count
  badge communicates the same information at a fraction of the layout cost.
  Open melds (face-up claimed tiles) should still render at readable size since they
  carry strategic information.
  **Files affected:** `GameTable2D.tsx`, new `MobileOpponentStrip2D.tsx`, possibly
  `Table2DContext.tsx`.

### BUG-2D-08 — Mobile: top status bar too tall, steals gameplay space

**Status:** OPEN
**Screenshot evidence (portrait):** The status bar renders with two full `MahjongTile2D`
spirit chips at what appears to be `sm` size (~36×48 px rendered at 1× scale), plus
the `节` label below each chip. The bar is approximately 70–80 px tall. Additionally
the wall count displays ALL remaining tiles as individual red squares in a horizontal
strip (visible in landscape screenshots as two rows of ~10 red squares at the top) —

**Root cause:**

1. `JingTileChip` renders `<MahjongTile2D size="xs" />` (28×38 px) but the `节`
   label and button padding push the chip taller than the surrounding text rows.
2. The wall tile visualisation (red squares at top in screenshots) appears to be
   rendered as part of the `OpponentHand2D` for the top seat — these are the top
   opponent's face-down tiles appearing as a horizontal strip just below the status
   bar, not a separate wall widget. This means the BUG-2D-07 top-opponent tiles ARE
   rendering — they land just below the status bar as a 10-tile strip of red squares.
   **Proposed fix:**

- Shrink `JingTileChip` to a 20×28 px thumbnail (a new `'2xs'` size or an explicit
  `style` override) with `py-0.5` button padding. Total bar height target: ≤ 36 px.
- The spirit tile "tap to enlarge" overlay already exists — the chip in the bar can
  be tiny since players can tap it whenever they need to see the full tile.
- Remove the `Wall NN` text label on mobile (move it to the enlarge overlay or keep
  only a number). Save the horizontal space for the spirit chip.
  **Files affected:** `game-page.tsx` (status bar, `JingTileChip`).

### BUG-2D-09 — Mobile: tiles too small / unreadable in the game table

**Status:** OPEN
**Screenshot evidence:** In portrait screenshots the right-opponent column shows
individual red tile backs as ~8×10 px squares — at this scale the Back SVG texture
is completely invisible; they appear as solid red rectangles. In landscape (screenshot
3–4) the opponent tile columns are still only ~12×16 px. The centre discard pool
is completely absent in portrait; in landscape partial tiles appear but are very
small. The viewer's own hand at the bottom renders full-size (~56×76 px `lg` tiles)
and looks correct on both orientations.
**Root cause:**

- `computeTileScale` with `oppAvail = tableH × 0.5 − 48` produces near-zero or
  negative values on portrait mobile (small center-row height → very small or
  clamped-to-MIN scale). The `TILE_SCALE_MIN = 0.25` floor means tiles can render
  at 7×9 px before the system stops shrinking — at which point tiles are invisible
  blobs, not readable mahjong tiles.
- There is no minimum _readable_ size constraint separate from the minimum _layout_
  constraint. The system allows tiles to reach pixel sizes that serve no visual
  purpose.
  **Proposed fix (tied to BUG-2D-07 mobile redesign):**
- With the compact badge layout for opponent face-down tiles, individual tile
  rendering only occurs for **open melds** (face-up, strategically important).
  Open meld tiles should render at a fixed minimum of `xs` size (28×38 px) regardless
  of scale.
- For the combined discard pool, enforce a minimum tile size of `sm` (36×48 px) and
  make the pool container **scrollable** rather than shrinking tiles to fit an 8-column
  fixed grid.
- Set `TILE_SCALE_MIN = 0.5` (14×19 px effective xs) as the absolute floor and make
  the pool scroll horizontally/vertically beyond that point.
  **Files affected:** `Table2DContext.tsx`, `CombinedDiscardPool2D.tsx`,
  `OpponentHand2D.tsx`, `OpenMelds2D.tsx`.

### BUG-2D-10 — Mobile: combined discard pool completely absent in portrait, partial in landscape

**Status:** OPEN
**Screenshot evidence:** In both portrait screenshots (11:05, 11:06) the centre green
felt area is completely empty even mid-game — no discard tiles are visible anywhere.
In landscape screenshots (11:07, 11:09) there appears to be one small tile near the
centre (the 片 tile visible in the compass area), which may be a mispositioned
discard tile rather than the combined pool rendering correctly.
**Root cause (confirmed):** The `center` CSS Grid cell has `gridArea:'center'` with
`position:relative` and `overflow:hidden`. In portrait mobile:

- The center row is `minmax(0, 2fr)` — but the grid's total height minus the
  `auto` bottom zone (viewer hand, ~100–130 px) and the status bar (~70 px) leaves
  very little height for the 1fr + 2fr rows to share.
- The center grid COLUMN is `1fr` between the two `22%` side columns — on a 375 px
  screen that is 375 - 2×82 = 211 px wide, which is correct.
- **The height problem:** The center row height may collapse near zero if the bottom
  `auto` zone consumes most of the remaining height. `overflow:hidden` then clips
  the absolutely-positioned `CombinedDiscardPool2D` completely.
- Even if the center cell has some height, the `top:50%; left:50%; transform:
  translate(-50%,-50%)` positioning anchors relative to the cell — if the cell is
  say 40 px tall, the pool centres within those 40 px, overflows, and is clipped.
  **Proposed fix (tied to BUG-2D-07):** In the mobile redesign, the discard pool
  should be positioned **absolutely relative to the full game container** (`position:
absolute; top:50%; left:50%`) rather than relative to a grid cell that can collapse.
  Remove `overflow:hidden` from `CenterDiscards2D`. Set `minHeight` on the center row
  (e.g., `minmax(120px, 2fr)`) as a stopgap for the current desktop layout so the
  pool always has a render area.
  **Files affected:** `CenterDiscards2D.tsx`, `CombinedDiscardPool2D.tsx`,
  `GameTable2D.tsx`.

## Key Learnings

- **Engine claim methods must patch two seats.** When a claim is resolved the claimed tile
  leaves the discarder's pile and enters the claimer's meld — the engine must update both
  seats atomically in one `withState` call using a two-seat patch. Patching only the
  claimer leaves a ghost entry in the discarder's `discards` array that propagates through
  every subsequent `toClientSnapshot` broadcast.

- **CSS rotateZ reverses DOM child visual order in a flex container.** For all three
  rotations used (180°, +90°, −90°) the DOM-FIRST flex child ends up at the visual
  INNER edge (facing the table centre) and the DOM-LAST child ends up at the visual
  OUTER edge (screen perimeter). This is the opposite of the intuitive reading order.
  When laying out a rotated opponent zone always put centre-facing content (hand, melds)
  first in the DOM and edge labels last.

- **CSS-rotated seat zone containers break the spatial meaning of "inside".** Rendering
  discard pools as children of a `rotateZ(±90°/180°)` seat zone makes them appear in the
  seat strip, not the felt centre. The fix is to lift the pools out of the rotated
  containers into a dedicated un-rotated overlay in the `center` grid area and position
  them absolutely per role. As a side effect, tile shadow offsets must be recalculated for
  the un-rotated context (use `role="bottom"` for all centre-area tiles rather than each
  seat's inverse-rotated offset). Similarly, column counts change: left/right pools that
  were 6-wide-and-visually-rotated become 3-wide-and-upright.
